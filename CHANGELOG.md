## Unreleased

- **File Management cross-filesystem target bridge and explorer** - Added a shared File Management target bridge for mounted volumes, scanned disk/partition inventory targets, and manual raw/image targets. The File Organizer now exposes the selected file-system target and blocks raw/non-native generic moves with explicit safety text; Duplicate Finder can scan supported ext2/ext3/ext4, HFS+/HFSX, and APFS raw/image targets read-only through existing S.A.K. readers; Advanced Search can search supported raw/image targets; and a new File Explorer tab provides target selection, path navigation, directory listing, copy path, text preview, and explicit create folder/write file/rename/delete actions for supported HFS+/HFSX and 64-128 MiB one-spaceman-chunk generated-layout APFS targets. Added the destructive `file_management_live_certifier` lane plus `scripts/run_file_management_live_filesystem_certification.ps1`; the public script now captures native certifier stderr under relaxed native-command error handling so informational worker logs do not false-fail the run. Current live proof covers APFS raw create/write/read/duplicate-search/advanced-search/delete/delete-folder on JMicron serial `DD56419883A5B`, Disk 2 Partition 2, and HFS+ raw create/write/read/duplicate-search/advanced-search/rename/delete/delete-folder on Best Buy serial `DD564198838A8`, Disk 3 Partition 3.
- **File Management Explorer Files-like foundation** - Added `FileExplorerTargetId`, normalized explorer locations, pane/tab state, view settings, selection summaries, and a UI-independent command registry. Target identity now requires bridge-provided explicit IDs and never derives synthetic IDs. The registry centralizes command text, shortcuts, destructive/write metadata, feature gates for tabs and dual pane, selection requirements, target capability checks, and exact raw write blockers.
- **File Management Explorer shell refactor** - Replaced the target-combo form layout with a Files-inspired shell: grouped sidebar target navigation, command bar, back/forward/up path row, editable path omnibar, model/view details table, bottom status strip, table/sidebar context menus, command shortcuts, and persistent Preview/Properties/Safety/Evidence tabs. Existing open, copy path, preview, create folder, write file, rename, and delete actions remain wired, while button/menu enabled states, status blockers, tooltips, and safety-pane text now use the shared command registry. The shell is now split into dedicated Qt widgets for sidebar, command bar, omnibar, main pane, and details pane so later M5-M12 work can evolve each area without re-growing `FileManagementExplorerPanel`. The sidebar now exposes Home, Favorites, This PC, Mounted Volumes, Disks and Partitions, Raw Images, Recent, and Certification Targets, with persisted favorites/recent/last-target state, target properties, text badges, and stale pane-target write guards. The details table now supports safe multi-select: rename/open/preview/properties stay single-item, while copy path and delete handle multiple selected items. Added `test_file_management_explorer_panel`, `test_file_explorer_item_model`, and expanded `test_file_explorer_types` coverage for shell creation, responsive collapse, sidebar groups, target selection, command blockers, omnibar load, context-menu actions, target-type command states, extended selection, and shortcut smoke coverage.
- **File Management Explorer details view hardening** - Added `FileExplorerSortFilterModel` and `FileExplorerDetailsView` for folder-first proxy sorting/filtering, attributes/capability summary column, persisted column widths/order, loading/empty/error state text, stale async listing result discard, and `QtConcurrent` directory listing so large local/raw target folders do not block the UI thread. GUI coverage now includes pane states, column persistence, and the open route for directory activation.
- **File Management Explorer Files parity correction** - Added a functional command-bar View/layout picker that switches Details/List/Grid/Cards/Adaptive views, exposes a Columns view with current-folder and lazy selected-directory child preview columns, controls item size, hidden items, and file-extension display, and persists view settings per target/path through the new File Explorer settings path only. View modes share one model, one selection model, and one command registry; unsupported future commands such as dual pane and new tab still show product-grade blockers. Search and command buttons execute real current-folder filtering and a registry-backed command palette instead of nonfunctional status-only handlers. The Files-like plan now explicitly moves deeper multi-level Columns polish, tabs/dual pane, and richer search ahead of preview polish so the explorer does not drift into a generic table UI.
- **File Management Explorer Files icon registry** - Added a `FileExplorerIconRegistry` backed by bundled MIT Files Community SVG assets for mapped command, layout, pane, Open, Copy Path, New Folder, Write File, Rename, Delete, Refresh, and dual-pane surfaces. The imported icons are traced to `files-community/Files` commit `8ea6d30` in `resources/icons/files/manifest.json`, registered in `THIRD_PARTY_LICENSES.md`, loaded through `resources/icons.qrc`, and covered by GUI tests that verify registry metadata and non-null toolbar/menu icons.
- **Partition Manager APFS generated-layout size guard** - Apple kernel validation in the Sonoma recovery VM rejected the previous 51 GB generated APFS raw target with `spaceman_sanity_check` checkpoint/container corruption even though the Windows-side verifier accepted it. Generated APFS format, repair, File Management writes, and Apply scripts now fail closed outside the certified one-spaceman-chunk 64-128 MiB envelope; the detector now ignores inconsistent APFS spaceman free-space counters when chunk/CIB geometry does not cover the container. Windows destructive APFS raw-format proof now passes on the 128 MiB JMicron slice at `artifacts/file-management-live-certification/disk2-apfs-128mb-raw-format/report.json`; Apple-native validation of that small slice remains pending.
- **Windows raw-device write alignment** - The shared raw-device `QIODevice` now handles unaligned logical writes with aligned read/patch/write cycles and flushes writable raw handles on close, fixing physical HFS+ catalog/data writes that previously failed on `GLOBALROOT` targets with `The parameter is incorrect`.

- **Partition Manager APFS object-type storage-class flags (found by Apple fsck_apfs)** - A macOS Sonoma recovery VM was stood up (VirtualBox, installer-media boot) to run Apple's own filesystem validators against S.A.K.-generated filesystems. `fsck_apfs -n` rejected the generated container with `o_type invalid, o_type 0x1 should be 0x80000001`: generated object headers carried bare type enums without APFS storage-class flag bits, which the Windows-side verifier had accepted. The generated-layout writer now emits proper storage classes - container superblock and space manager ephemeral (`0x80000000`), container/volume object maps and their B-trees physical (`0x40000000`), volume superblock and root file-system tree virtual - and the layout verifier expects the flagged values (the detector already masked flags). Apple's real `fsck_hfs` (hfs-650.0.2) in the same VM fully passed S.A.K.-built HFS+ volumes: catalog, extents overflow, multi-linked files, depth-3 catalog hierarchy, extended attributes, and volume bitmap. Continued fsck_apfs iterations caught and fixed, in order: `nx_next_oid` inside the reserved OID range; the missing checkpoint descriptor area (now blocks 10-17 with a checkpoint map and superblock copy); the missing ephemeral reaper (`nx_xp_data_len`); all dynamic OIDs below the 1024 reserved floor (volume/root-tree/spaceman/reaper now ≥1025, physical objects use their block address); and zeroed `nx_ephemeral_info` (now the canonical 0x800040001, byte-verified against a real Apple Sonoma container extracted host-side from BaseSystem.dmg). The generated space manager now carries real internals (chunk geometry, a chunk-info block, and an allocation bitmap). The generated layout was then rebuilt to mirror a same-geometry `newfs_apfs` container position-for-position (two-checkpoint genesis, ghost objects scheduled on real free-queue records, chunk-info/bitmaps inside the internal pool, ghost container omap, empty extent-ref and snapshot-metadata trees, full volume-superblock field set including `nx_next_doc_id` and the unencrypted-volume flag), reaching byte parity with Apple's output everywhere except UUIDs and the root file-system tree. Apple's checker now passes the checkpoint, space manager, free-queue trees, both object maps, the volume superblock, and the snapshot metadata; a guest-written reference file provided the record-level ground truth for the final gate, and the rebuilt record engine closed it: **`fsck_apfs` now reports both the generated volume and the generated container as "appears to be OK", and the macOS kernel auto-mounts the generated volume.** The record engine writes Apple's base entities (tree-root entity with `root`/`private-dir` children), hashed directory-entry keys (CRC-32C over UTF-32LE code points, verified against three Apple-written samples), dstream-id records, full inode values (parent/private ids, timestamps, 0x8000 flags, NAME and DSTREAM extended fields with Apple's exact `xf_used_data` semantics), block-rounded extent lengths, file ids numbered from 16, B-tree info flags/longest-key/value fields, and key ordering by numeric name-hash. A reader order-dependency (extended-field offsets computed in absolute rather than value-relative coordinates) and a directory/file object-id collision in the rewrite path were found and fixed along the way. The remaining cosmetic fsck warning was then root-caused and eliminated: the extent-reference tree now carries a physical-extent record per file (key `paddr | PHYSEXT`, value `{block_count|kind, owning inode, refcount}`), after which `fsck_apfs` reports zero warnings on single- and multi-block files. Arbitrary Apple-media APFS mutation shipped via the new `import-image` flow: any unencrypted, snapshot-free container that the generic reader can walk — including containers written by Apple's own APFS driver — is read file-by-file and re-emitted through the certified writer with optional added/overwritten files; certified in the VM by round-tripping and mutating a kernel-written container with clean `fsck_apfs` verdicts. The HFS+ journal replay was additionally validated against an authentic crash journal: a `newfs_hfs -J` volume written by macOS was hard-reset mid-transaction in the VM, the dirty image captured before remount, S.A.K.'s `replayJournalFromImage` applied the open transaction (1 transaction, 4 blocks, alternate-header sync), and Apple's forced `fsck_hfs -fn` then reported the volume "appears to be OK" with macOS mounting it read-write.
- **Partition Manager HFS+ attribute lifecycle completion** - New `deleteAttributeValue` removes inline and fork-backed extended attributes (releasing allocated blocks and the free-block counter exactly, freeing the attributes B-tree leaf when it empties, and clearing `kHFSHasAttributesMask` on the owning catalog record when the last attribute is removed), and new `createForkAttributeValue` creates fork-backed attributes with fresh block allocations. CLI commands `delete-attribute-image` and `create-fork-attribute-image`; real-volume self-test proof with `fsck_hfs -n -f` after every create/delete step.
- **Partition Manager HFS+ decmpfs LZVN/LZFSE codecs** - decmpfs types 7/8 (LZVN) and 11/12 (LZFSE) now decompress on read and compress on replacement via Apple's reference lzfse/lzvn library vendored at `third_party/lzfse` (BSD 3-Clause, registered in THIRD_PARTY_LICENSES.md), including the bare little-endian offset-table resource-fork container and the LZVN raw-chunk escape. Unknown types keep failing closed.
- **Partition Manager HFS+ three-level catalog trees** - The catalog mutation engine rebuilds index levels generically: depth-3 trees (root index over index nodes over leaves, with per-level sibling links) mutate, split up from depth 2, and collapse back down; the working bound is 256 leaves and 3 levels. A depth-3 unit scenario drives two root splits and a full drain back to one level; the CLI self-test proves depth-3 bulk create (new `create-empty-files-image` command) and delete on a real `newfs_hfs` volume with clean `fsck_hfs` after each phase.
- **Partition Manager HFS+ journal replay** - New `replayJournal` core API and `replay-journal-image` CLI command replay dirty little-endian in-filesystem HFS+ journals: journal-info-block/header/transaction checksum and geometry validation, circular-buffer transaction reads, sector-addressed block writes under the configured write cap, and a verified clean-header commit. Certified by byte-equality against the certified direct-write path plus fail-closed proofs (tampered transaction checksum, non-journaled volume, big-endian journal, out-of-volume targets); `newfs_hfs -J` needs-init journals no-op cleanly.
- **Partition Manager HFS+ two-level catalog engine** - The catalog writer is now a unified tree-mutation engine for one- and two-level catalog B-trees: every catalog mutation (create, delete, allocated delete, folder-tree delete, rename/move) loads the working tree (header map, root index, all leaves; bounded to 64 leaves), applies per-leaf edits with root-index key maintenance, splits overflowing leaves at either depth, removes emptied leaves, collapses two-level trees back to one level when a single leaf remains, and erases freed nodes with exact free-node/map/leaf-count accounting. Depth-3 trees and full root-index splits stay fail-closed. Coverage: a depth-2 unit scenario (third-leaf split, first-leaf index-key updates, cross-leaf rename, folder lifecycle, full drain with collapse), CLI self-test post-split create/rename/delete on a real `newfs_hfs` volume with `fsck_hfs -n -f` after each phase, and physical raw proof with staged depth-2 create/attribute-create/delete plus raw read-back hashes (run 2026-06-10 UTC).
- **Partition Manager HFS+ alternate volume header sync** - Every header-mutating HFS+ operation now copies the primary volume header to the alternate volume header (1024 bytes before end of volume), replacing the prior "alternate volume header was not updated" warning class with a synchronized proof line.
- **Partition Manager HFS+ attribute create** - New `createInlineAttributeValue` core API and `create-inline-attribute-image` CLI command create brand-new inline extended attributes: the attributes B-tree gains empty-tree first-leaf materialization, single-leaf insert, and root-leaf split, and the owning catalog record's `kHFSHasAttributesMask` flag is set so `fsck_hfs` attribute-count buckets stay consistent. Duplicate names, missing catalog IDs, missing attributes forks, and non-opt-in `com.apple.decmpfs` writes fail closed. Real-volume CLI proof materializes the attributes tree on a `newfs_hfs` volume, passes `fsck_hfs`, and verifies the created value hash through the probe certifier; the physical lane proves attribute create on raw USB HFS media.
- **Partition Manager HFS+ B-tree split engine** - `PartitionHfsFileSystemWriter` now performs a catalog B-tree root-leaf split when a create operation no longer fits a one-level catalog: it allocates three free nodes from the catalog header-node map, builds new left/right leaf nodes plus a new root index node as orphan writes, commits with a single header-node write that updates tree depth, root node, first/last leaf, leaf-record count, free-node count, and map bits, then erases the freed old root leaf so `fsck_hfs` accepts the volume. Split preconditions (big keys, variable-length index keys, enough free nodes inside the header-node map, consistent allocation map) fail closed, and catalog mutation on the resulting two-level tree stays blocked with a structured `single leaf node` blocker. Created thread records are now sized to the actual node name instead of a maximum-length record, matching Apple's on-disk layout. Coverage: unit fixtures for empty-file/data-file/folder splits plus fail-closed variants, a CLI self-test sequence on a real `newfs_hfs` volume validated by bundled `fsck_hfs -n -f` after the split, and physical raw-partition proof on pinned expendable USB HFS media.
- **Partition Manager HFS+ extents-overflow growth engine** - Broad allocation growth and bounded file create beyond eight extents now keep the first eight extents in the catalog record and insert overflow records (eight extents per record, keyed by file ID, fork type, and logical start block) into the extents-overflow B-tree, with empty-tree first-leaf materialization, single-leaf insert/remove, root-leaf split, tree free on last-record removal, and freed-node erasure. Allocated-file delete and folder-tree delete remove the deleted forks' overflow records in the same operation. Coverage: unit fixtures for overflow growth/delete and extents root-leaf split, a CLI self-test that fragments a real `newfs_hfs` volume, grows a file into overflow records, passes `fsck_hfs` after growth and after delete, and verifies payload hashes through the probe certifier, plus physical raw proof with raw overflow-file read-back and staged raw overflow-delete on pinned expendable USB HFS media at `artifacts\partition-manager-certification\vm-lab\external-evidence\external.hfs-file-apply-physical\report.json`.
- **Partition Manager HFS+ decmpfs compressed-file round trips** - `readFile` now transparently decompresses `com.apple.decmpfs` type-3 (inline zlib or raw-marker) and type-4 (resource-fork chunked zlib) compressed files with read-cap enforcement, and the new `replaceCompressedFileContent` writer API plus `replace-compressed-image` CLI command rewrite compressed content in the same compression type behind the explicit compressed-file mutation opt-in with decompression read-back proof. Type-4 resource forks are rebuilt with the documented 64 KiB chunk table and resource-map trailer layout. decmpfs types 7/8 (lzvn) and 11/12 (lzfse) fail closed with structured blockers. Compressed replacement is exposed at the core-API and helper-CLI level only (same exposure model as generated-image APFS root-file writes); it is not a Partition Manager panel action until a dedicated UI milestone adds its own queue operation, safety validation, and Apply review. Coverage: unit round trips for both types plus unsupported-type and no-opt-in blockers, and a CLI fail-closed proof against non-compressed targets.
- **Partition Manager APFS snapshot/encryption structural pinning** - The generated-layout verifier used by every generated-layout APFS mutation route now pins snapshot, revert, and encryption state to zero: the volume superblock must carry no snapshot-metadata tree OID, no revert XID/superblock OID, zero snapshots, and zero volume flags, and both the container and volume object maps must carry no snapshot or pending-revert state. Tampered images fail closed with explicit snapshot/revert/encryption blockers. The read-only detector's APFS object-map snapshot field offsets were corrected to the on-disk `omap_phys` layout (`om_tree_oid` 0x30, snapshot tree OID 0x38, most-recent snapshot XID 0x40, pending revert 0x48/0x50), so snapshot-risk gating now reads the real fields. Coverage: unit tamper tests for all five pinned states; large physical raw APFS destructive lane evidence is retained as Windows-side only until rerun inside the 64-128 MiB one-spaceman-chunk envelope and Apple-validated.
- **Partition Manager panel UX** - Disk inventory no longer starts on panel display. Linux swap raw detection now parses signature page size, header version, last page, bad-page count, UUID, and label. HFS+/HFSX raw detection now parses direct and classic-wrapper volume-header version, journal flag, file/folder counts, block size, and total/free byte counts when the header's block math is valid. Raw detector metadata is persisted into the partition table tooltip and Properties dialog, an Inspect Non-Windows File System sidebar/context action opens a copyable metadata report, a Check Non-Windows File System action opens an original read-only metadata consistency report for XFS/Btrfs/APFS when captured sanity data exists, Check Non-Windows File System also opens an original read-only HFS+/HFSX catalog consistency and attributes B-tree key scan report, and a Browse Non-Windows File System action lists ext2/ext3/ext4, HFS+/HFSX, and APFS directory entries, extracts selected regular files, and exports ext/HFS+/APFS folders recursively through original read-only parser code without mounting or writing source volumes. Added `docs/PARTITION_MANAGER_CROSS_FILESYSTEM_PLAN.md`, `PartitionFileSystemRegistry`, `PartitionExtFileSystemReader`, `PartitionHfsFileSystemReader`, `PartitionApfsFileSystemReader`, `PartitionFileSystemToolManifest`, `PartitionFileSystemToolRunner`, `scripts/check_partition_filesystem_tool_manifest.ps1`, and a CMake-required `tools/filesystem/manifest.json` approval gate for portable Linux/HFS+ detection, read, format, and repair support using original code or bundled open-source tools with no runtime installs. The app now runs as a standard user and elevates only for specific tasks that require it, via an elevated helper process communicating over Named Pipes IPC. Tasks drop back to standard privileges once complete.
- **Elevation tier system** - Introduced `ElevationTier` (Standard/Elevated/Mixed) and `FeatureId` enums classifying 40+ features. Consolidated three duplicate `isRunningAsAdmin()` implementations into `ElevationManager::isElevated()`.
- **Elevated helper process** - New `sak_elevated_helper.exe` with `requireAdministrator` manifest, connected via Named Pipes with a custom framed protocol (`[4-byte LE length][1-byte type][UTF-8 JSON]`). Security: random nonce in pipe name, parent PID validation, DACL restrictions, task allowlist, single connection.
- **Elevation broker** - `ElevationBroker` client manages launching the helper, sending task requests, receiving results, and handling cancellation/timeout. Integrated into `QuickActionController` for seamless elevated action execution.
- **Elevation gates** - Reusable `showElevationGate()` dialog prompts users before elevated operations (Image Flasher, UUP ISO, App Installation). Users can restart elevated or decline.
- **Mixed-tier operations** - Backup worker path-access detection (`canReadPath()`), `PermissionManager` `std::expected` overloads, 5 new helper task handlers (TakeOwnership, StripPermissions, SetStandardPermissions, BackupFile, ReadThermalData), `ThermalMonitor::hasCpuTemperature()`.
- **Status bar elevation indicator** - Icons8 Flat Color SVG icons (shield for admin, lock for standard user) with text label and tooltip. "Run as Admin" button to restart elevated when needed.
- **Elevation info banners** - Informational banner on Image Flasher and App Installation panels indicating that some operations may prompt for elevation.
- **Legacy cleanup** - Removed `runElevatedQuickAction()`, `requestAdminElevation()`, CLI argument parsing (`--run-quick-action`), and associated Win32 headers. No more ShellExecuteEx-based self-re-launch.
- **New error codes** - `elevation_required`, `elevation_failed`, `elevation_denied`, `elevation_timeout`, `helper_connection_failed`, `helper_crashed`, `task_not_allowed`.
- **New tests** - 6 new test files with 139 test methods covering elevation tiers, IPC protocol, task dispatcher, mixed-tier operations, UX components, and hardening verification.
- **Comprehensive documentation accuracy audit** - Verified all in-app About panel feature descriptions, wizard welcome pages, README highlights, CHANGELOG entries, and build script claims against the actual codebase. All feature descriptions confirmed accurate. Corrected header file count (142 headers including action headers), corrected README test count reference (74 automated tests).
- **Build quality** - Clean MSVC `/W4 /WX` build, 93 automated tests (all passing), 157 source files, 167 headers.

## v0.9.0.8

- **Offline deployment - Direct Download mode** - Downloads the actual installer binary (EXE/MSI) for each package instead of just the .nupkg wrapper. Supports three package patterns: URL-based downloads (`Install-ChocolateyPackage`), embedded-binary packages (`Install-ChocolateyInstallPackage` with EXE/MSI files inside the nupkg's `tools/` directory), and meta-packages (follows `.install` dependency chain via nuspec parsing). Primary installer selection ensures only the main resource is downloaded per package, avoiding patches and secondary resources. Tested with Office PC preset (10 packages, all succeeding).
- **Install script parser improvements** - Splatting regex now matches any variable name (not just `$packageArgs`) with line-anchored closing brace to handle `${variable}` syntax in URLs. `extractHashtableValue` uses two-pass parsing (quoted first, unquoted fallback). `resolveVariables` handles both `$varName` and `${varName}` references.
